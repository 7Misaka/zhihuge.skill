## 1. 使用方式

先做结构审计，再做句子审计。对每项标记 `PASS`、`PARTIAL`、`FAIL` 或 `N/A`，并附文件位置和修复动作。存在任一关键 `FAIL` 时，不输出“论文已形成完整闭环”。

关键失败包括：主贡献无证据、定理假设缺失、主要比较不公平、数值来源不明、结论超出实验范围。

## 2. 来源与可追溯性

- [ ] 论文题目、版本和链接来自原始来源。
- [ ] 所有具体数值可定位到表、图、正文或运行产物。
- [ ] 所有定理条件与原始符号一致。
- [ ] 二手材料只用于发现，不作为核心方法事实的唯一依据。
- [ ] 跨论文归纳明确标为归纳，不冒充作者原话。
- [ ] 无法核实的内容标为待确认，未被补造。

## 3. 叙事闭环

- [ ] 一句话贡献包含问题、原因、机制、理论、实验和边界。
- [ ] 宏观问题已拆成可处理的机制子问题。
- [ ] 每个研究问题至少由一个组件、理论结果或实验回答。
- [ ] 每个组件都能追溯到具体问题。
- [ ] 每项贡献都同时写出动作、目标、机制和证据。
- [ ] 摘要、引言、贡献列表和结论使用同一组核心主张。
- [ ] 相关工作最后落到本文的精确缺口。
- [ ] 讨论处理了最重要的极端情形或反直觉结果。

## 4. 章节完整性

### 摘要

- [ ] 前半段从场景快速收缩到具体失败。
- [ ] 方法句解释了组件为何有效。
- [ ] 理论句说明对象和关键依赖，而非只写“提供理论分析”。
- [ ] 实验句有范围和指标，数字均有来源。
- [ ] 没有正文未定义的缩写或新贡献。

### 引言

- [ ] 每段只有一个主功能。
- [ ] 缺口是机制性或证据性的，不是泛泛“研究不足”。
- [ ] 研究问题可由正文实际回答。
- [ ] 方法概览映射到研究问题。
- [ ] 贡献条目相互正交。

### 相关工作

- [ ] 分类依据与问题、组件或理论工具一致。
- [ ] 每条路线同时交代能力和限制。
- [ ] 最近、最接近工作得到直接比较。
- [ ] 不使用无证据的“首次”“唯一”或全面优越性措辞。

### 问题定义与方法

- [ ] 符号、目标、图、数据和更新顺序完整。
- [ ] 基线失败发生在哪一步写清楚。
- [ ] 每个组件的输入、操作、输出和成本完整。
- [ ] 公式前有目的，公式后有逐项解释。
- [ ] 算法框与正文、理论和代码使用同一顺序。

### 讨论、结论与附录

- [ ] 讨论没有复制结果段，而是在解释原因和边界。
- [ ] 结论没有引入新主张。
- [ ] 局限说明影响哪条结论。
- [ ] 附录顺序与正文依赖一致。
- [ ] 正文引用附录时说明附录提供的内容。

## 5. 理论完整性

- [ ] 理论目标与论文主张相匹配。
- [ ] 定义和假设在使用前给出。
- [ ] 每个强假设说明用途、常见程度和失效情形。
- [ ] 技术困难相对既有证明被明确指出。
- [ ] 桥梁引理连接算法机制和最终理论对象。
- [ ] 主定理声明对象、口径、条件和依赖关系。
- [ ] 每一界项有语义解释。
- [ ] 与基线的比较使用相同假设和指标。
- [ ] 推论给出可解释的参数选择和方向性预测。
- [ ] 极限分析没有被误写为有限样本必然结论。
- [ ] 泛化主张由泛化或稳定性对象支持，而非训练损失替代。
- [ ] 证明覆盖所有新增状态和更新，不沿用不适用的旧引理。

## 6. 实验完整性

- [ ] 每个核心主张至少有一个直接实验或理论证据。
- [ ] 每个主实验明确回答某个 RQ。
- [ ] 数据、模型、节点、异质性、拓扑和预算定义完整。
- [ ] 基线足够接近且调参预算公平。
- [ ] 报告随机种子、重复次数和不确定性。
- [ ] 区分训练轮数、通信量、计算量和墙钟时间。
- [ ] 组件消融单独移除每个贡献组件。
- [ ] 超参数扫描控制其他变量。
- [ ] 拓扑实验报告可量化网络属性。
- [ ] 理论验证提前写出预测方向和否证条件。
- [ ] 报告无优势、失败、饱和或冲突设置。
- [ ] 经验相关性未被写成理论证明或因果识别。

## 7. 逐句质量

对摘要、引言首尾句、贡献列表、定理解释和结果结论逐句执行：

- [ ] 该句只有一个主要言语功能。
- [ ] 主语明确，避免连续使用模糊的“this/it/其”。
- [ ] 因果词后有机制或设计支持。
- [ ] 比较词有对象、指标和条件。
- [ ] 强度词与证据等级匹配。
- [ ] 数值包含单位、口径和基准。
- [ ] 句中符号已定义。
- [ ] 该句与前句形成推进，而非同义改写。
- [ ] 段末句完成结论、边界或过渡。
- [ ] 删除该句后若论证不受影响，考虑删去。

## 8. 过度声称

出现以下词时强制检查：`prove`、`guarantee`、`optimal`、`robust`、`generalize`、`efficient`、`significant`、`state-of-the-art`、`first` 及其中文对应词。

### 8.1 强度匹配表

| 证据状态 | 可用措辞 | 禁用措辞 |
|---|---|---|
| 正式定理且条件满足 | 保证、上界、收敛到 | 无条件保证、所有场景 |
| 多设置受控实验 | 一致观察到、在所测设置中改善 | 证明、普遍最优 |
| 单一或探索性实验 | 表明、提示、与预测一致 | 显著优越、稳健 |
| 机制解释无直接识别 | 可能由于、与…相容 | 导致、证明机制 |
| 证据缺失 | 假设、待验证 | 任何确定性结论 |

### 8.2 边界检查

- [ ] “鲁棒”说明了对什么扰动或变化鲁棒。
- [ ] “高效”说明了相对哪个成本口径。
- [ ] “可扩展”覆盖了规模变化而不只是一个大模型。
- [ ] “泛化更好”报告测试或泛化对象。
- [ ] “收敛更快”区分理论阶、轮数和墙钟时间。
- [ ] “显著”有统计含义时提供检验，否则换词。

## 9. 复现与投稿前检查

- [ ] 算法伪代码可由正文独立实现。
- [ ] 超参数搜索空间和选择规则可复现。
- [ ] 数据划分与异质性构造可复现。
- [ ] 拓扑生成、混合矩阵和连通性检查可复现。
- [ ] 图表可追溯到原始运行或数据文件。
- [ ] 表中最好/次好标记由程序或一致规则生成。
- [ ] 正文数字与表图一致。
- [ ] 匿名、伦理、许可和资源声明符合目标 venue 当前要求。
- [ ] 页数压缩没有删除关键假设、对照或边界。
- [ ] 附录与代码版本对应当前稿件。

## 10. 缺陷分级与交付格式

### 10.1 分级

- `P0`：事实、数字、定理或实验来源错误；阻止交付。
- `P1`：核心主张—证据断链、比较不公平、关键假设遗漏；阻止“完成”声明。
- `P2`：章节职责混乱、机制解释不足、复现细节缺失；应在提交前修复。
- `P3`：措辞、过渡、局部冗余；不改变结论但影响清晰度。

### 10.2 审计输出

| Priority | Location | Failed item | Evidence | Required fix |
|---|---|---|---|---|
| P1 |  |  |  |  |

最后输出：

1. `结论`：通过 / 有条件通过 / 不通过。
2. `核心断链`：最多五项，按 P0→P3 排序。
3. `不可声称内容`：当前证据不允许写出的结论。
4. `最小修复路径`：使主张闭环所需的最少动作。
5. `剩余边界`：修复后仍需在论文中保留的限制。
