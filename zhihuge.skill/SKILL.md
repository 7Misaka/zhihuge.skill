# 智虎哥理论驱动论文框架

把论文写成一条可检验的因果链：**问题 → 机制 → 组件 → 定理 → 实验 → 边界**。不要先堆章节或术语；先锁定主张以及每项主张所需的证据，再展开文本。

## 核心原则

1. 只让一句话承担一个主要功能：定义、判断、因果、对比、证据或过渡。
2. 把宏观痛点拆成两个可由不同机制处理的子问题；每个方法组件只解决一个明确子问题。
3. 让理论揭示变量如何影响结果，而不只给出“收敛”或“可泛化”的结论。
4. 让实验直接操纵理论中的变量，并检验定理给出的方向性预测。
5. 把强假设、极端情形、反直觉发现和无效场景写进讨论或局限，不要藏在附录。
6. 区分来源事实、跨论文归纳和当前论文设计建议；不得把归纳写成原作者的直接主张。

## 可复用论文框架（唯一规范）

后续生成章节、段落和句子时，严格采用本节的章节数量、句子数量和句子功能。不要自行扩写为另一套框架。

### 0. 写作前的主张—证据矩阵

正文动笔前，每个核心贡献填写一行：

| 主张 | 问题现象 | 原因机制 | 方法组件 | 理论项 | 受控实验 | 边界 |
|---|---|---|---|---|---|---|
| C1 | 什么指标恶化 | 为什么恶化 | 哪个组件处理 | 哪个定理项解释 | 改变什么变量 | 何时不成立 |
| C2 |  |  |  |  |  |  |
| C3 |  |  |  |  |  |  |

任何空白都代表论文尚未闭环：

- 有方法、无理论：只能写经验贡献。
- 有理论、无实验：不能宣称已得到经验验证。
- 有总体性能、无组件消融：不能声称某个组件导致提升。
- 只有上界：不能将上界写成实际性能的精确等式。

### 1. 标题

推荐结构：

- 方法型：`[方法名]: [核心机制] for [任务或问题]`
- 结果型：`[动作性结果]: [分析方法] in [研究场景]`
- 问题型：`Why/Can [技术现象]? A [理论视角]`
- 机制型：`Unveiling/Understanding [技术机制]: [理论贡献]`

标题必须同时出现：

1. 研究对象。
2. 核心机制或理论视角。
3. 主要结果或问题。

### 2. Abstract：9—10 句

#### 通用九句结构

1. **领域价值句**：说明研究范式解决什么现实约束。
2. **现有方案句**：说明当前主流方法如何工作。
3. **失败现象句**：指出现有方案在特定条件下出现什么可观测问题。
4. **研究缺口句**：说明该问题尚未被算法或理论真正解决。
5. **方法总览句**：提出方法或分析框架，并说明总体目标。
6. **组件一机制句**：写第一个组件处理哪个原因。
7. **组件二机制句**：写第二个组件处理哪个原因。
8. **理论结果句**：给出优化、稳定性或泛化结果，保留关键阶数和变量。
9. **实验结果句**：写数据集、场景、比较对象和最重要的定量结果。
10. **边界或意义句，可选**：理论分析型论文写无法消除的极限差距；算法型论文写实际部署意义。

#### 摘要检查

摘要中的每个结果都必须能在正文定位到：

- 一个定理、推论或 Remark。
- 一张主表或主图。
- 一个明确的实验设置。

### 3. Introduction：六段，约 18 句

#### 第 1 段：领域与范式，3 句

1. 定义研究领域及现实需求。
2. 说明主流范式的工作方式和优势。
3. 引出目标范式为什么值得研究。

#### 第 2 段：问题的因果链，4 句

4. 写目标范式的第一个可观测问题。
5. 解释导致该问题的中间机制。
6. 写第二个问题及其机制。
7. 说明两个问题最终如何影响性能、效率或泛化。

不要只写“性能较差”，应写成：

> 在 `[条件]` 下，`[机制变量]` 持续累积，导致 `[中间现象]`，最终使 `[指标]` 恶化。

#### 第 3 段：研究缺口或研究问题，2—3 句

8. 总结已有方法仍未处理的具体部分。
9. 明确提出研究问题一。
10. 明确提出研究问题二。

算法型论文可以把问题写成设计目标；机制解释型论文最好直接使用问句。

#### 第 4 段：方法与组件映射，4 句

11. 提出方法或理论框架。
12. 说明组件一修改了哪个更新阶段、解决哪个问题。
13. 说明组件二修改了哪个阶段、解决哪个问题。
14. 解释两个组件为何互补，而不是简单堆叠。

#### 第 5 段：证据预览，4 句

15. 概述理论对象：优化、稳定性、泛化或极限分析。
16. 写最重要的理论结论以及关键变量。
17. 概述实验范围和比较对象。
18. 写最重要的定量提升或理论趋势验证。

#### 第 6 段：贡献列表

每条贡献采用同一格式：

1. **做了什么**。
2. **解决哪个缺口**。
3. **通过什么机制**。
4. **由什么理论或实验支撑**。

推荐顺序：

- 方法或分析框架贡献。
- 理论和机制贡献。
- 实验与实际指导贡献。
- 边界或极限结论，可选。

### 4. Related Work：三条研究线

开头用一句话声明将回顾哪些研究线。

每条研究线采用六句模板：

1. 定义该研究线与本文问题的关系。
2. 介绍早期代表方法解决了什么。
3. 介绍近期方法如何改进。
4. 指出最接近本文的方法及其机制。
5. 分析这些方法仍依赖什么假设或遗漏什么变量。
6. 用缺口句结束，并把缺口连接到本文贡献。

推荐三条线：

- 任务或范式：DFL、分布式优化等。
- 方法机制：ADMM、SAM、Momentum、Catalyst、MGS 等。
- 理论工具：收敛、稳定性、泛化、拓扑分析等。

不要按作者年份流水账排列。每个小节都必须以“尚缺什么”结束。

### 5. Background / Problem Setup：10 句

这部分可以独立成章，也可以作为 Methodology 的第一小节。

1. 定义参与方、迭代轮数和局部步骤。
2. 定义数据分布及异质性来源。
3. 定义通信图、邻居集合和混合矩阵。
4. 定义总体或 population objective。
5. 定义 empirical objective。
6. 定义全局最优值、经验最优值和算法输出。
7. 定义需要分析的误差量。
8. 给出基线算法的核心更新式。
9. 指出基线更新式中产生问题的具体项。
10. 说明后续方法或分析将如何处理该项。

理论分析型论文还应增加：

- 稳定性定义。
- 泛化误差与优化误差分解。
- 将稳定性连接到泛化误差的桥接引理。

### 6. Methodology

#### 6.1 章节导语：2 句

1. 说明本节先定义什么、再设计什么。
2. 说明完整方法由哪几个组件组成。

#### 6.2 每个组件：七句单元

对组件一和组件二分别写：

1. **失败定位**：基线中的哪个步骤导致什么问题。
2. **设计目标**：本组件需要控制什么量。
3. **数学修改**：给出修改后的目标函数或更新式。
4. **符号解释**：解释新增变量、正则项或控制量。
5. **机制解释**：说明它如何改变优化轨迹或模型状态。
6. **问题回链**：指出它对应引言中的哪个问题。
7. **代价与边界**：说明新增计算、通信、超参数或假设。

多组件关系再写三句：

1. 单独使用组件一能解决什么，不能解决什么。
2. 单独使用组件二能解决什么，不能解决什么。
3. 两者联合为何互补，并对应哪个组合消融实验。

#### 6.3 完整算法：4 句

1. 说明算法输入、输出和初始化。
2. 说明每轮局部更新的执行顺序。
3. 说明通信、聚合或外推顺序。
4. 指明关键公式分别对应算法的哪些行。

### 7. Theoretical Analysis

#### 7.1 章节导语：3 句

1. 本节分析哪个算法输出和误差。
2. 先给哪些定义与假设。
3. 随后给哪些定理，以及证明放在哪里。

#### 7.2 每个假设：三句

1. 正式写假设。
2. 解释该假设控制什么现象。
3. 说明它是否标准、能否放松，以及后续哪里使用。

不要把“有界梯度”和“有界随机梯度噪声”混为一谈。

#### 7.3 分析难点：4 句

1. 标准分析为何不能直接使用。
2. 新算法中的哪个项破坏了标准递推。
3. 引入什么辅助序列、稳定性量或桥接条件。
4. 该工具如何把复杂更新转成可分析对象。

#### 7.4 桥接引理：4 句

1. 引理连接哪两个量。
2. 写正式结论。
3. 解释右侧各项的含义。
4. 说明控制哪个量即可得到最终结果。

#### 7.5 每个主定理：八句

1. 写适用假设。
2. 写参数范围。
3. 写定理结论。
4. 拆解主界中的优化项。
5. 拆解异质性、噪声或拓扑项。
6. 给出简化阶数。
7. 翻译成参数变化预测。
8. 指向验证该预测的实验小节。

#### 7.6 每个 Remark：五句

1. 明确讨论哪个变量。
2. 从定理中提取包含该变量的项。
3. 判断该变量变化造成的方向。
4. 给出实际参数选择或系统设计建议。
5. 写明该结论的条件或非单调边界。

#### 7.7 极限与比较：六句

1. 令关键变量趋于极端值。
2. 写极限界。
3. 写理想方法或中心化方法的界。
4. 比较二者关于样本、节点或轮数的标度。
5. 判断差距能否被单一机制消除。
6. 把该边界加入摘要、讨论和结论。

### 8. Experiments

#### 8.1 实验导语：3 句

1. 说明要验证的三类主张。
2. 说明比较范围和主要场景。
3. 指出附录包含哪些实现细节。

#### 8.2 Experiment Setup：12 句

1. 数据集及其任务。
2. 模型架构。
3. IID 设置。
4. 非 IID 或困难程度构造。
5. 客户端数量。
6. 参与率。
7. 通信拓扑。
8. 中心化基线。
9. 去中心化或同类基线。
10. 训练轮数、批量和学习率。
11. 方法特有超参数。
12. 计算量和通信量的公平性说明。

#### 8.3 主性能结果：七句

1. 指出表格回答的研究问题。
2. 陈述整体结果。
3. 写最重要的绝对数值。
4. 写相对最强基线的提升。
5. 比较不同问题强度下的变化。
6. 解释结果与哪个机制一致。
7. 回链到贡献和主张编号。

#### 8.4 收敛或效率：六句

1. 定义固定性能阈值。
2. 报告达到阈值所需轮数或时间。
3. 计算相对加速比。
4. 说明是否同时节省通信或计算。
5. 对照理论中的轮数或参数项。
6. 说明速度提升是否牺牲最终性能。

#### 8.5 环境或拓扑实验：六句

1. 定义各环境或拓扑条件。
2. 说明它们改变了哪个理论量。
3. 报告不同条件下的性能。
4. 判断是否符合理论预测。
5. 讨论反直觉结果。
6. 给出部署层面的建议。

#### 8.6 每个超参数消融：七句

1. 说明参数在算法中的作用。
2. 从定理中指出它影响的项。
3. 写扫描范围。
4. 说明其他变量保持不变。
5. 报告最佳区间。
6. 解释过小和过大分别为何失败。
7. 判断实验是否支持理论预测。

#### 8.7 组件消融：六句

1. 定义完整模型。
2. 移除所有新增组件。
3. 仅保留组件一。
4. 仅保留组件二。
5. 同时使用两个组件。
6. 判断二者是互补、冗余还是存在交互。

#### 8.8 理论趋势验证：七句

1. 选择能代理理论对象的观测量。
2. 说明如何构造受扰动或配对数据。
3. 改变一个理论关键变量。
4. 绘制经验趋势。
5. 与理论阶数或方向比较。
6. 报告不吻合的区域。
7. 说明这种验证支持的是趋势，不是定理证明。

### 9. Discussion：8 句

1. 提出最重要的反直觉发现。
2. 说明已有理论通常给出什么结论。
3. 说明本文理论为何不同。
4. 给出相应实验观察。
5. 区分收敛性能与泛化性能。
6. 解释关键超参数为何有效。
7. 说明极限情况和不可消除的差距。
8. 给出实际使用建议和开放问题。

当理论和实验表面冲突时，Discussion 不能省略。

### 10. Conclusion and Limitation：8 句

1. 重述研究问题，而不是重复背景。
2. 概括方法或分析框架。
3. 说明组件一解决什么。
4. 说明组件二解决什么。
5. 概括最重要的理论结论。
6. 概括最重要的实验结论。
7. 写理论假设、规模或未验证条件。
8. 给出具体未来方向。

结论不能引入正文未验证的新贡献。

### 11. Appendix

建议顺序：

1. 实验实现和超参数。
2. 通信与计算配置。
3. 补充曲线和数据集结果。
4. 符号表。
5. 关键引理。
6. 优化定理证明。
7. 稳定性或泛化证明。
8. 批量、拓扑或极限扩展分析。
9. 额外局限性讨论。

### 最终复用原则

- 引言中的问题数量，应与方法组件的逻辑数量大致对应。
- 每个理论变量都应转化成一项实验干预。
- 每项实验都要能回链到引言中的一个主张。
- 理论贡献不只给阶数，还要给参数选择、极限和不可达结论。
- 摘要和结论只报告已经进入“主张—机制—理论—实验”闭环的内容。

## 工作流

### 1. 建立来源账本

优先读取原始论文全文。记录题目、版本、章节、图表、定理、实验变量和局限。若输入只有摘要或二手材料，明确降低结论置信度，不补造方法细节、数值或作者意图。

需要复现本技能的三篇来源与交叉归纳时，读取 [cross-paper-evidence.md](references/cross-paper-evidence.md)。

### 2. 判断论文类型

选择最接近的主路线；混合型论文可合并，但必须指出主次。

- **A：新算法型**：突出双问题、双组件、优化理论、性能与消融。
- **B：算法 + 双理论型**：在 A 的基础上同时证明优化和泛化，并增加独立讨论节。
- **C：机制解释型**：用明确研究问题驱动背景、稳定性或误差桥梁、极限分析与理论验证实验。

### 3. 锁定一句话贡献

用一句话填满以下槽位：

> 现有方法因【机制性原因】在【场景】中产生【可观察问题】；本文通过【核心机制】控制【关键变量】，并以【理论对象】和【受控实验】证明【结果与边界】。

若句中任何槽位为空，先补研究设计，不开始润色。

### 4. 建立主张—证据矩阵

为每项贡献分配唯一 ID，并填写：观察到的问题、因果机制、方法组件、理论对象、受控变量、指标、预期方向、边界与落稿位置。使用 [claim-evidence-matrix.md](references/claim-evidence-matrix.md) 的模板和双向审计规则。

### 5. 设计研究问题与组件

把一句话贡献拆成 2–4 个研究问题。优先采用下面的映射：

- RQ1 解释或控制核心优化问题；组件 C1 与定理 T1 回答。
- RQ2 解释或控制泛化、异质性或拓扑问题；组件 C2 与定理 T2 回答。
- RQ3 检查联合机制、极端情形或适用边界；消融、敏感性和讨论回答。

禁止添加无法追溯到研究问题的组件，也禁止提出没有理论或实验回答的研究问题。

### 6. 生成章节与逐句蓝图

严格使用上文“可复用论文框架（唯一规范）”的章节和句子槽位，按论文类型裁剪。先输出每段的功能，再输出句子槽位；句子槽位写“应包含的事实和证据”，不要提前生成空泛成文。除非用户明确要求扩写，否则不得改变规定句数。

### 7. 绑定理论与实验

对每个核心定理执行以下动作：

1. 明示假设和被界定对象。
2. 把界拆成优化项、网络项、数据项、异质性项或稳定性项。
3. 从界中抽取可操纵变量及方向性预测。
4. 为每个预测配置单变量实验、指标和控制条件。
5. 用结果回答定理预测是否成立，而不把相关性写成证明。

### 8. 写讨论、局限和结论

优先讨论谱隙、通信步数、局部步数、学习率、节点数、样本量、异质性等极端条件。区分“定理保证”“实验观察”“机制解释”和“待验证猜想”。结论只闭环主问题、机制、理论、实验和边界，不引入新贡献。

### 9. 完成审计

使用 [quality-checklist.md](references/quality-checklist.md) 逐项检查。发现断链时回到矩阵修复，不通过改写句式掩盖证据缺口。

## 标准交付格式

按顺序输出：

1. 论文类型与判断依据。
2. 一句话贡献。
3. 研究问题列表。
4. 主张—证据矩阵。
5. 章节树及每章唯一职责。
6. 每节段落功能。
7. 每段逐句槽位，标记 `事实`、`推理`、`证据`、`过渡` 或 `边界`。
8. 组件—研究问题映射。
9. 定理—变量—实验映射。
10. 风险、缺失证据和不可声称内容。
11. 最终自检结果。

若用户只要求其中一项，只交付所需部分，但仍在内部完成主张—证据检查。

## 写作约束

- 使用可证伪的机制语言，避免“有效提升”“充分证明”等无对象、无条件表述。
- 给最近工作写出“做到什么、仍缺什么、本文补什么”，不要罗列文献。
- 贡献列表保持正交；每条同时包含动作、目标、机制和证据。
- 方法节解释更新顺序、状态变量、通信位置和新增计算；理论节不替代方法节。
- 主结果、消融、敏感性、拓扑分析和理论验证各回答不同问题。
- 不虚构定理、数据集、基线、超参数、显著性或数值。缺失内容标为“待作者提供”。
- 不把跨论文稳定结构机械套用到纯系统、纯数据集或纯应用论文；先说明需要改造的章节。

## 不适用边界

不要把本框架当作固定 LaTeX 模板、自动文献事实生成器或实验结果替代品。对纯综述、纯数据论文、无理论目标的工程系统论文，仅借用“主张—证据”和逐句功能设计，不强制添加定理。对具体会议格式、页数和匿名政策，另行读取目标 venue 的最新官方要求。
